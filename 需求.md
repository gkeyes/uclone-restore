# UClone Restore 0.3 重建需求基线

This document defines behavior for the rebuild that starts from the proven `v0.2.0` implementation. Requirements not listed here are not release blockers for 0.3.

## Product boundary

- The main App runs in the main Android user and performs all Root, backup, restore, rollback, notification, and task-history work.
- The LSPosed module only injects the Launcher entry and submits authenticated requests.
- Default users are main `user0` and clone `user10`, but runtime settings remain authoritative.
- UClone never copies `/data/app` or `/data/misc/keystore`.
- The stable 0.2 switch, restore, push, clone unlock, and module cold-start behavior is the compatibility baseline.

## Data directions

```text
switch:  current user10 data -> user0
restore: saved user0 return point -> user0
push:    current user0 data -> user10
```

Switch does not update user10. Push does not change the user0 switch marker.

## Backup classes

- Active snapshot: a user-managed restore source.
- Persistent MAIN backup: the fixed return point created from known main-system data. Normal switching never replaces an existing valid return point.
- Legacy persistent CLONE backup: a labeled copy created by older builds. It remains readable and user-managed, but normal switching never selects or creates it.
- Transaction undo: the exact target state captured immediately before a live App data mutation. It is never replaced by a persistent backup.
- Clone rollback: the exact target protection created before a push into the clone user.

The first successful MAIN-to-CLONE switch creates the fixed MAIN return point. Later switches keep that return point unchanged. Every MAIN-to-CLONE switch still creates a fresh transaction undo of current user0; the fixed MAIN return point never replaces that protection. A later temporary MAIN undo is deleted after a verified successful commit.
The fresh undo is mandatory until data and marker commit. Only the normal MAIN-to-CLONE switch may initialize a missing fixed MAIN return point from that fresh undo. Generic restore and rollback operations cannot initialize or replace it. If marker publication fails after directory promotion, the promotion is reverted before automatic rollback. A pre-existing fixed return point is never used for failure rollback during the live MAIN-to-CLONE mutation.

Normal CLONE switching always reads current user10 data. A legacy persistent CLONE backup is not the live clone source.
If an old switch marker references `persistent_clone` or `persistent_clone.previous`, state detection returns `UNKNOWN`; normal MAIN restoration must not select that legacy CLONE backup.

## App state

- `MAIN`: user0 is known to contain main-system data.
- `CLONE`: user0 is known to contain clone-system data and has a valid MAIN return point.
- `UNKNOWN`: the data source cannot be proven.

UI labels use this shared result. Deleting logs, refreshing pages, or resetting the displayed state must not fabricate `MAIN`.

## Workspace ownership

- New workspace payload is extracted without archive owner restoration and must be owned by `root:root`.
- Real App restores still apply the target App UID/GID and SELinux context.
- Existing workspace ownership repair is a user-confirmed maintenance task. It never runs during startup or list refresh and never deletes data.

## Permission and AppOps policy

- Permission and AppOps migration is best-effort in 0.3.
- A capture failure is different from a valid empty capture.
- Capture failure skips that permission component with a warning; it does not revoke permissions, reset AppOps, or abort valid file-data copying.
- No ROM-specific command fallback becomes production behavior before its exact command result is captured on the target device.
- The 0.3 rebuild preserves the proven 0.2 cache and OBB owner behavior until target-device metadata proves a replacement. AOSP formulas or audit conclusions alone do not authorize a production change.
- On the current Android 16 HyperOS target, `cmd appops get --user <user> <package>` contains UID-scope entries followed by package-scope entries, while `cmd appops get --user <user> <uid>` returns the UID scope. UClone 0.3 preserves UID-scope AppOps and migrates only a positively identified package scope.
- `cmd package list packages --user <user> <missing-package>` may return exit code 0 with empty output. Package presence is therefore determined only by an exact `package:<name>` record, never by exit code alone.

## Device verification contract

- ROM-dependent behavior is specified from recorded target-device commands, not inferred from command names, exit codes, AOSP behavior, or host tests.
- A regression test preserves each confirmed command/output contract, but does not replace phone acceptance.
- Data-flow changes are complete only after the exact fixed-signed GitHub artifact is installed and the relevant operation passes on the target phone.
- The acceptance record includes installed App/module versions, request ID, full task log, final notification/history status, MAIN/CLONE/UNKNOWN marker, rollback path, and observed source/target App data.
- Confirmed target-command evidence is recorded in `docs/DEVICE_EVIDENCE_2026-07-13.md`; its "Not yet proven" section remains binding until replaced by newer target-device evidence.
- Before that phone acceptance, the build remains device-unverified and must not replace the stable 0.2 release.

## Cross-user installation

- `install-existing` enables code already installed for another Android user.
- Modes are install only, install plus best-effort permissions, and install plus data sync.
- Installation failure prevents sync. Sync failure after installation is partial success and does not uninstall the package.
- UClone itself is always blocked. System packages are not offered in the first 0.3 implementation.

## Switch safety modes and copy-pass contract

- `SAFE` is the default for new and migrated installations. `DANGEROUS_FAST` is an explicit global opt-in that affects only a known `CLONE` to `MAIN` restore.
- A copy pass means one complete selected-data tree write. Metadata scans, permission capture, and failure-only rollback do not count as normal-path passes.
- MAIN-to-CLONE is identical in both modes: `current user0 MAIN -> fresh temporary undo`, then `current user10 CLONE -> user0`. The normal successful path is two copy passes.
- SAFE CLONE-to-MAIN is `current user0 CLONE -> temporary checkpoint`, `checkpoint -> user10`, then `fixed MAIN return point -> user0`. The normal successful path is three copy passes. The checkpoint is the exact user0 undo and push source; it is deleted only after both targets and the MAIN marker commit.
- DANGEROUS_FAST CLONE-to-MAIN is `current user0 CLONE -> user10`, then `fixed MAIN return point -> user0`. The normal successful path is two copy passes and intentionally has no local CLONE checkpoint.
- Neither mode creates a second full copy of the old user10 state during this composite restore. The standalone manual push operation keeps its existing exact clone rollback behavior.
- Every selected data part preserves its source state exactly: `data` copies content, `empty` clears the target directory, and `absent` removes the target directory. Checkpoints record an internal `unselected` state for excluded parts; those parts are neither copied nor restored.
- If user10 synchronization fails, MAIN restore does not start and user0 remains `CLONE`. If user10 was already mutated, the task reports that the clone target may need resynchronization.
- In SAFE mode, a controlled MAIN restore failure rolls user0 back from the temporary CLONE checkpoint. In DANGEROUS_FAST mode, the verified user10 copy is the only surviving current CLONE data; 0.3 does not automatically copy it back to user0, and a failed MAIN restore leaves the marker `UNKNOWN` for manual recovery.
- Both stages share one request ID, task, notification, and history record.
- 0.3 protection covers controlled shell failures only. Persistent execution gates, process-death/reboot recovery, and durable transaction journals remain 0.4 work, so UI text must not claim crash or power-loss safety.

The fixed MAIN return point can only be replaced by an explicit main-App action after UClone has written an explicit confirmed-MAIN marker through a successful MAIN restore. A legacy missing marker is not sufficient proof for this destructive replacement. The replacement is prepared and validated before the previous return point is moved aside; a normal commit failure restores the previous directory. This operation is not exposed to the module or launcher shortcut.

## 0.4 boundary

Persistent transaction journals, App execution gates, startup recovery, cooperative cancellation, and a durable `RECOVERY_REQUIRED` state are 0.4 work. Version 0.3 only emits a fatal task/log marker for controlled partial-mutation failures; it does not persist or recover that state across process death or reboot.
