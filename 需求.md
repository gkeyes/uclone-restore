# UClone Restore 0.3 重建需求基线

This document defines behavior for the rebuild that starts from the proven `v0.2.0` implementation. Requirements not listed here are not release blockers for 0.3.

## Product boundary

- The main App runs in the main Android user and performs all Root, backup, restore, rollback, notification, and task-history work.
- The LSPosed module only injects the Launcher entry and submits authenticated requests.
- Default users are main `user0` and clone `user10`, but runtime settings remain authoritative.
- UClone never copies `/data/app` or `/data/misc/keystore`.
- The stable 0.2 switch, restore, push, clone unlock, and module cold-start behavior is the compatibility baseline.

## Data directions

```text
switch:  current user10 data -> user0
restore: saved user0 return point -> user0
push:    current user0 data -> user10
```

Switch does not update user10. Push does not change the user0 switch marker.

## Backup classes

- Active snapshot: a user-managed restore source.
- Persistent MAIN backup: the return point created from known main-system data. Its update policy is either fixed or refresh-on-MAIN-exit.
- Legacy persistent CLONE backup: a labeled copy created by older builds. It remains readable and user-managed, but normal switching never selects or creates it.
- Transaction undo: the exact target state captured immediately before a live App data mutation. It is never replaced by a persistent backup.
- Clone rollback: the exact target protection created before a push into the clone user.

The first successful MAIN-to-CLONE switch creates the MAIN return point. `FIXED` keeps an existing valid return point unchanged. `REFRESH_ON_MAIN_EXIT` atomically promotes the fresh transaction undo to the new return point when user0 has an explicit confirmed-MAIN marker; this reuses the mandatory undo and does not add a copy pass. If an existing return point is present but MAIN is only inferred from a missing legacy marker, switching may continue but automatic refresh is skipped with a warning. An invalid existing return point always blocks switching before target mutation.
Every MAIN-to-CLONE switch creates a fresh transaction undo of current user0. The undo is mandatory until data and marker commit and is always the controlled-failure rollback source. If return-point promotion or marker publication fails, the previous return point and transaction undo are restored before rollback. Generic restore and rollback operations cannot initialize or replace the return point.

Normal CLONE switching always reads current user10 data. A legacy persistent CLONE backup is not the live clone source.
If an old switch marker references `persistent_clone` or `persistent_clone.previous`, state detection returns `UNKNOWN`; normal MAIN restoration must not select that legacy CLONE backup.

## App state

- `MAIN`: user0 is known to contain main-system data.
- `CLONE`: user0 is known to contain clone-system data and has a valid MAIN return point.
- `UNKNOWN`: the data source cannot be proven.

UI labels use this shared result. Deleting logs, refreshing pages, or resetting the displayed state must not fabricate `MAIN`.

## Workspace ownership

- New workspace payload is extracted without archive owner restoration and must be owned by `root:root`.
- Real App restores still apply the target App UID/GID and SELinux context.
- Existing workspace ownership repair is a user-confirmed maintenance task. It never runs during startup or list refresh and never deletes data.

## Permission and AppOps policy

- Permission and AppOps migration is best-effort in 0.3.
- A capture failure is different from a valid empty capture.
- Capture failure skips that permission component with a warning; it does not revoke permissions, reset AppOps, or abort valid file-data copying.
- No ROM-specific command fallback becomes production behavior before its exact command result is captured on the target device.
- The 0.3 rebuild preserves the proven 0.2 cache and OBB owner behavior until target-device metadata proves a replacement. AOSP formulas or audit conclusions alone do not authorize a production change.
- On the current Android 16 HyperOS target, `cmd appops get --user <user> <package>` contains UID-scope entries followed by package-scope entries, while `cmd appops get --user <user> <uid>` returns the UID scope. UClone 0.3 preserves UID-scope AppOps and migrates only a positively identified package scope.
- `cmd package list packages --user <user> <missing-package>` may return exit code 0 with empty output. Package presence is therefore determined only by an exact `package:<name>` record, never by exit code alone.

## Device verification contract

- ROM-dependent behavior is specified from recorded target-device commands, not inferred from command names, exit codes, AOSP behavior, or host tests.
- A regression test preserves each confirmed command/output contract, but does not replace phone acceptance.
- Data-flow changes are complete only after the exact fixed-signed GitHub artifact is installed and the relevant operation passes on the target phone.
- The acceptance record includes installed App/module versions, request ID, full task log, final notification/history status, MAIN/CLONE/UNKNOWN marker, rollback path, and observed source/target App data.
- Confirmed target-command evidence is recorded in `docs/DEVICE_EVIDENCE_2026-07-13.md`; its "Not yet proven" section remains binding until replaced by newer target-device evidence.
- Before that phone acceptance, the build remains device-unverified and must not replace the stable 0.2 release.

## Cross-user installation

- `install-existing` enables code already installed for another Android user.
- Modes are install only, install plus best-effort permissions, and install plus data sync.
- Installation failure prevents sync. Sync failure after installation is partial success and does not uninstall the package.
- UClone itself is always blocked. System packages are not offered in the first 0.3 implementation.

## Switch data policies and copy-pass contract

- `MainReturnPointPolicy` is `FIXED` or `REFRESH_ON_MAIN_EXIT`; the default is `FIXED`.
- `CloneSessionPolicy` is `SYNC_TO_CLONE_USER` or `DISCARD_ON_MAIN_RETURN`; the default is `SYNC_TO_CLONE_USER`.
- `SwitchSafetyMode` is `SAFE` or `DANGEROUS_FAST`; the default is `SAFE`.
- All three policies are global settings snapshotted when a task is admitted. Main App, desktop shortcut and module requests use the same snapshot.
- A copy pass means one complete selected-data tree write. Metadata scans, permission capture, and failure-only rollback do not count as normal-path passes.
- MAIN-to-CLONE is always `current user0 MAIN -> fresh transaction undo`, then `current user10 CLONE -> user0`: two copy passes. Refreshing MAIN promotes the undo by directory move after validation rather than copying it again.
- `SYNC_TO_CLONE_USER + SAFE` is `current user0 CLONE -> temporary checkpoint`, `checkpoint -> user10`, then `MAIN return point -> user0`: three copy passes.
- `SYNC_TO_CLONE_USER + DANGEROUS_FAST` is `current user0 CLONE -> user10`, then `MAIN return point -> user0`: two copy passes and no local checkpoint.
- `DISCARD_ON_MAIN_RETURN + SAFE` is `current user0 CLONE -> temporary rollback`, then `MAIN return point -> user0`: two copy passes. The rollback is never written to user10 and is deleted only after MAIN data and marker commit.
- `DISCARD_ON_MAIN_RETURN + DANGEROUS_FAST` is `MAIN return point -> user0`: one copy pass. It intentionally discards the current user0 CLONE session and has no local rollback.
- Both discard plans must not start, unlock, stop, inspect or mutate user10, including permissions and AppOps.
- Neither mode creates a second full copy of the old user10 state during this composite restore. The standalone manual push operation keeps its existing exact clone rollback behavior.
- Every selected data part preserves its source state exactly: `data` copies content, `empty` clears the target directory, and `absent` removes the target directory. Checkpoints record an internal `unselected` state for excluded parts; those parts are neither copied nor restored.
- In sync plans, if user10 synchronization fails, MAIN restore does not start and user0 remains `CLONE`; partial user10 mutation is reported explicitly.
- In SAFE plans, a controlled MAIN restore failure rolls user0 back from the temporary CLONE checkpoint. In FAST plans, a failure after user0 mutation leaves the marker `UNKNOWN` for manual recovery. Discard plans never claim that user10 was updated or damaged.
- Both stages share one request ID, task, notification, and history record.
- All four combinations may be triggered by the main App, desktop shortcut, or module after the global settings have been saved. Main-App confirmations and external notifications must state the exact plan and copy-pass count.
- 0.3 protection covers controlled shell failures only. Persistent execution gates, process-death/reboot recovery, and durable transaction journals remain 0.4 work, so UI text must not claim crash or power-loss safety.

The MAIN return point can be replaced either by the explicit main-App update action or by `REFRESH_ON_MAIN_EXIT` after an explicit confirmed-MAIN marker. A legacy missing marker is not sufficient proof for replacing an existing return point. Every replacement is prepared and validated before the previous return point is moved aside; a normal commit failure restores the previous directory. The explicit update action remains unavailable to the module and launcher shortcut.

## Restore performance contract

- Active snapshots, MAIN passive backups, and clone rollback backups are managed, immutable restore sources. Normal restore reads their exact workspace path directly, does not create a second `tmp/prepared_*` payload, and post-commit retention must not delete the selected source. A managed source root and its selected top-level data parts must be real directories, not symbolic links; only task-created live switch parts may be links and those are resolved explicitly for sizing.
- Direct source use changes only the read path. It does not change MAIN/CLONE direction, selected data parts, permission policy, UID/GID handling, SELinux handling, durability barriers, markers, or task admission.
- A fresh transaction undo of the current target remains mandatory before ordinary restore mutation. Direct source use must never replace, reuse, or skip that undo.
- Restore source validation counts each selected source tree once. A successful target copy is checked with a non-empty probe; a second full target count or `du` is not required unless a live switch removes `cache` or `code_cache` and therefore changes the item count.
- The preflight target size is reused only as an in-task transaction-undo estimate. Completed managed backups may publish strict per-part numeric hints under `.size_kb`; JSON manifest sizes and pre-stop transaction estimates are never authoritative restore inputs. Direct restore sums only the selected parts and requires free space for the larger of that source sum and the current target. A missing, multiline, malformed, symlinked, or otherwise invalid hint falls back to one read-only `du` scan for that part; any failed size scan must stop before target mutation rather than silently reduce the requirement to zero.
- `UCLONE_PERF` lines are diagnostic output only. They must not change task status, persisted `TaskMetrics`, notifications, history, or external request results.
- The normal-path copy-pass contracts above remain unchanged. Performance work must reduce redundant scans or temporary copies, not remove a safety copy pass.
- Restore performance acceptance follows `docs/RESTORE_PERFORMANCE_ACCEPTANCE.md`. Phone evidence from the exact fixed-signed GitHub artifact is required before the optimization is considered deployable.

## 0.4 boundary

Persistent transaction journals, App execution gates, startup recovery, cooperative cancellation, and a durable `RECOVERY_REQUIRED` state are 0.4 work. Version 0.3 only emits a fatal task/log marker for controlled partial-mutation failures; it does not persist or recover that state across process death or reboot.
