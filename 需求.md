# UClone Restore 0.3 重建需求基线

This document defines behavior for the rebuild that starts from the proven `v0.2.0` implementation. Requirements not listed here are not release blockers for 0.3.

## Product boundary

- The main App runs in the main Android user and performs all Root, backup, restore, rollback, notification, and task-history work.
- The LSPosed module only injects the Launcher entry and submits authenticated requests.
- Default users are main `user0` and clone `user10`, but runtime settings remain authoritative.
- UClone never copies `/data/app` or `/data/misc/keystore`.
- The stable 0.2 switch, restore, push, clone unlock, and module cold-start behavior is the compatibility baseline.

## Data directions

```text
switch:  current user10 data -> user0
restore: saved user0 return point -> user0
push:    current user0 data -> user10
```

Switch does not update user10. Push does not change the user0 switch marker.

## Backup classes

- Active snapshot: a user-managed restore source.
- Persistent MAIN backup: a labeled reusable copy of known main-system data.
- Persistent CLONE backup: a labeled reusable copy of known clone-system data.
- Transaction undo: the exact target state captured immediately before a mutation. It is never replaced by a persistent backup.
- Clone rollback: the exact target protection created before a push into the clone user.

The "reuse persistent state backup" option may skip creation of an already validated MAIN or CLONE persistent copy. It must never skip the fresh transaction undo required for the current mutation.

## App state

- `MAIN`: user0 is known to contain main-system data.
- `CLONE`: user0 is known to contain clone-system data and has a valid MAIN return point.
- `UNKNOWN`: the data source cannot be proven.

UI labels use this shared result. Deleting logs, refreshing pages, or resetting the displayed state must not fabricate `MAIN`.

## Workspace ownership

- New workspace payload is extracted without archive owner restoration and must be owned by `root:root`.
- Real App restores still apply the target App UID/GID and SELinux context.
- Existing workspace ownership repair is a user-confirmed maintenance task. It never runs during startup or list refresh and never deletes data.

## Permission and AppOps policy

- Permission and AppOps migration is best-effort in 0.3.
- A capture failure is different from a valid empty capture.
- Capture failure skips that permission component with a warning; it does not revoke permissions, reset AppOps, or abort valid file-data copying.
- No ROM-specific command fallback becomes production behavior before its exact command result is captured on the target device.
- The 0.3 rebuild preserves the proven 0.2 cache and OBB owner behavior until target-device metadata proves a replacement. AOSP formulas or audit conclusions alone do not authorize a production change.
- On the current Android 16 HyperOS target, `cmd appops get --user <user> <package>` contains UID-scope entries followed by package-scope entries, while `cmd appops get --user <user> <uid>` returns the UID scope. UClone 0.3 preserves UID-scope AppOps and migrates only a positively identified package scope.
- `cmd package list packages --user <user> <missing-package>` may return exit code 0 with empty output. Package presence is therefore determined only by an exact `package:<name>` record, never by exit code alone.

## Device verification contract

- ROM-dependent behavior is specified from recorded target-device commands, not inferred from command names, exit codes, AOSP behavior, or host tests.
- A regression test preserves each confirmed command/output contract, but does not replace phone acceptance.
- Data-flow changes are complete only after the exact fixed-signed GitHub artifact is installed and the relevant operation passes on the target phone.
- The acceptance record includes installed App/module versions, request ID, full task log, final notification/history status, MAIN/CLONE/UNKNOWN marker, rollback path, and observed source/target App data.
- Confirmed target-command evidence is recorded in `docs/DEVICE_EVIDENCE_2026-07-13.md`; its "Not yet proven" section remains binding until replaced by newer target-device evidence.
- Before that phone acceptance, the build remains device-unverified and must not replace the stable 0.2 release.

## Cross-user installation

- `install-existing` enables code already installed for another Android user.
- Modes are install only, install plus best-effort permissions, and install plus data sync.
- Installation failure prevents sync. Sync failure after installation is partial success and does not uninstall the package.
- UClone itself is always blocked. System packages are not offered in the first 0.3 implementation.

## Forced clone-data refresh

- The setting is off by default and applies only when a known `CLONE` state is restored to `MAIN`.
- The composite operation is `push current user0 clone-state data to user10 -> restore the saved MAIN return point to user0`.
- Both stages share one request ID, task, notification, and history record.
- If push fails, MAIN restore does not start and the state remains `CLONE`.

## 0.4 boundary

Persistent transaction journals, App execution gates, startup recovery, cooperative cancellation, and `RECOVERY_REQUIRED` are 0.4 work. They must be introduced one independently device-tested mechanism at a time and are not copied wholesale from the abandoned 0.3 branch.
