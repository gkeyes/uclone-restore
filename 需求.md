# UClone Restore 需求基线

> 本文档是 UClone Restore 与 UClone Launcher Module 的需求源。实现、测试、界面文案和发布说明必须与本文档一致。如需改变本文档中的不可变规则，必须先与用户确认。

## 1. 产品目标

UClone Restore 是一款 Android Root 多用户 App 数据工具。主 App 安装和运行在主系统用户，通过 Root 访问主系统与分身系统的 App 数据，完成：

- 从分身系统建立主动快照。
- 把分身数据切换或恢复到主系统。
- 把主系统当前数据单向推送到分身系统。
- 在覆盖目标数据前保存可回滚的事务前镜像。
- 在主系统和分身系统之间启用已存在的 App 安装。
- 通过 LSPosed 模块在桌面 App 图标长按菜单中提供快捷操作入口。

## 2. 边界与默认假设

- 主系统默认为 `user0`，分身系统默认为 `user10`。
- 两个用户 ID 必须是不同的非负整数，且必须在设备上存在。
- 主 App 不依赖分身内的 App 进程或分身内 Root。
- 所有真实数据任务由 UClone 主 App 执行；模块只负责菜单注入、请求转发和状态展示。
- 不复制 `/data/app`，跨用户安装使用 `install-existing`。
- 不复制 `/data/misc/keystore`。使用 Android Keystore 的登录态可能无法完整迁移。
- 默认排除 `cache` 和 `code_cache`，保留 `app_webview`和 `no_backup`。
- UClone 自身禁止同步、安装和模块控制。系统 App 默认禁止安装或模块控制。

## 3. 数据部件精确语义

每个 App 数据由以下独立部件组成：

| 部件 | 源路径示例 | 默认 |
|---|---|---|
| CE | `/data/user/<userId>/<package>` | 开 |
| DE | `/data/user_de/<userId>/<package>` | 开 |
| external | `/data/media/<userId>/Android/data/<package>` | 开 |
| media | `/data/media/<userId>/Android/media/<package>` | 关 |
| OBB | `/data/media/<userId>/Android/obb/<package>` | 关 |
| 权限/AppOps | PackageManager 与 AppOps 状态 | 开 |

快照或事务备份必须对每个被选择的文件部件记录一种状态：

- `excluded`：规则明确不处理。恢复时不能修改目标部件。
- `absent`：源路径不存在。恢复时必须删除目标路径。
- `empty`：源路径存在但没有内容。恢复时必须保留目标根目录并清空其内容。
- `data`：源路径存在且有内容。恢复时必须用快照内容精确替换目标内容。
- `unreadable`：路径应当可用但读取失败。任务必须在停止或修改目标 App 之前失败。

强制规则：

- 快照不得用“没有子目录”替代部件状态。
- `absent` / `empty` 状态下不得同时存在非空 payload。
- `data` 状态下必须存在非空 payload。
- 快照中任何被选中部件状态缺失时，快照视为损坏，禁止恢复。

## 4. CE 与分身生命周期

- 需要读取分身 CE 数据时，只有 `RUNNING_UNLOCKED` 允许继续。
- 如分身未启动或已锁定，且“分身自动解锁”开关开启，数据任务可发起非阻塞启动、传入已保存凭据并轮询 CE 状态。
- `am start-user`、`am stop-user` 与 `get-started-user-state` 的客户端返回不得作为用户生命周期完成的唯一依据；必须以受限时的实际用户状态查询为准，异步发起请求并及时终止、回收或脱离已经卡住但状态已达标的 ActivityManager 客户端。状态查询客户端超时后必须立即失败，不得继续解锁或复制数据。
- 轮询用于及时发现状态变化，不得以固定 10 秒、30 秒或 120 秒的无条件延时代替状态检测。
- 解锁密码只能通过受保护 stdin 传入 Root shell，不得出现在命令文本、日志或任务记录中。
- “无感启动分身”是持久生命周期操作：启动或解锁后不自动关闭分身。
- 数据任务仅在同时满足以下条件时可在结束后请求关闭分身：
  1. “任务后关闭分身”开关开启；
  2. 分身是本次数据任务从未启动状态启动的；
  3. 没有并发任务正在使用分身。
- 显式“启动”、“进入”、“无感启动”不自动关闭；只有显式“关闭分身”可无条件请求关闭。
- 不使用 `am start-user -w` 或长时间固定 sleep。状态等待必须是短周期轮询并设置有界上限。

## 5. 备份类型与保留规则

### 5.1 主动快照

- 由用户主动创建，默认来自分身系统。
- 当前恢复源位于 `snapshots/<package>/active`。
- 更新 active 时，原 active 移入 `history/<id>`，不与被动回滚备份混用。
- 删除 active 只删除主动快照，不删除被动备份、切换状态或日志。

### 5.2 持久状态备份

- “主数据”备份保存主系统原始状态。
- “分数据”备份保存分身系统原始状态。
- “已有备份不重复生成”开关仅允许复用已验证完整的持久状态备份，且必须按主数据/分数据区分。
- 清理任务日志不影响备份存在性或切换状态。

### 5.3 事务回滚点

- 每次要修改目标 App 数据时，必须在修改前生成能代表“本次任务开始瞬间”的事务回滚点。
- 事务回滚点与持久状态备份是两个概念。旧的持久备份不得直接作为当前任务失败后的自动回滚源。
- 如“不重复生成持久备份”开启，仍必须生成当前事务的临时 undo 镜像；任务成功后可清理 undo，任务失败时用 undo 恢复。
- 自动回滚失败必须是致命失败，不得降级为“成功但有警告”。

## 6. 操作定义

| 操作 | 源 | 目标 | 必要行为 |
|---|---|---|---|
| 建立主动备份 | 分身 | UClone active | 要求源 CE 已解锁；不修改主系统 App |
| 切换到分身数据 | 分身当前数据 | 主系统 | 先保存主数据，再精确覆盖，成功后记录主数据返回点 |
| 还原主系统数据 | 切换前的主数据备份 | 主系统 | 恢复成功后才清除切换状态 |
| 恢复指定备份 | 用户选择的已知数据源 | 对应用户 | 恢复后根据备份的 `stateKind` 更新状态；旧备份来源不明时标记未知 |
| 推送主数据到分身 | 主系统当前数据 | 分身 | 先保存分数据，再精确覆盖；不改变主系统切换按钮状态 |
| 仅安装到另一侧 | 系统已有 APK | 另一用户 | 不解锁 CE，不复制数据 |
| 安装并迁移权限 | 源用户 | 另一用户 | 安装成功后仅恢复经验证可支持的权限/AppOps |
| 安装并同步数据 | 源用户 | 另一用户 | 安装成功后进入现有数据事务；同步失败不自动卸载 |

### 6.1 强制更新分数据

- 设置页提供“强制更新分数据”开关，默认关闭。
- 开关只作用于 App 已处于 `CLONE` 状态、用户执行“还原主系统数据”的路径。
- 开启后，必须先把当前主系统用户（默认 `user0`）中正在使用的分身态数据同步回分身用户（默认 `user10`），完整成功后才能读取切换前主数据备份并还原主系统用户。
- 更新分数据必须复用“推送主数据到分身”的完整事务保护，包括源/目标 App 门禁、分身回滚、权限处理、空间预检、自动回滚和任务日志，不得降级为直接复制目录。
- 更新分数据失败时，必须停止整个组合任务；不得恢复主数据、不得清除切换标记，主系统继续保持 `CLONE` 状态。
- 更新分数据成功但后续主数据还原失败时，必须保留前一阶段已提交的分数据，并按主数据还原事务的结果进入自动回滚或 `RECOVERY_REQUIRED`。
- 该开关不影响首次“切换到分身数据”、手动“推送到分身”、普通被动备份恢复、主动快照恢复和分身回滚恢复。
- 同一个 requestId、Root 任务锁、通知和任务历史必须覆盖整个“更新分数据 -> 还原主数据”组合操作，UI 不得显示成两个互不相关的任务。

## 7. 状态机

主系统 App 数据状态必须是：

- `MAIN`：已知为主系统状态。
- `CLONE`：已知由分身数据恢复而来，且有有效的主数据返回点。
- `UNKNOWN`：用户手动修改数据、恢复旧的未标识备份、重置状态或检测到标记与数据不一致。

规则：

- UI 不得自己推导状态，必须使用统一状态解析结果。
- 首页的“切换/还原”是同一个主操作。`MAIN` 显示“切换”，`CLONE` 显示“还原”，`UNKNOWN` 必须要求用户进入详情确认。
- 只有恢复主数据返回点成功后才能从 `CLONE` 转为 `MAIN`。
- 删除当前主数据返回点默认禁止。用户必须先完成还原，或在高级操作中明确放弃返回能力并进入 `UNKNOWN`。
- “重置状态”不得伪造 `MAIN`。它只能将状态设为 `UNKNOWN`，且不修改 App 数据。

## 8. 包身份与恢复兼容性

每个新快照和备份 manifest 必须保存：

- packageName
- 来源用户与目标用户
- 来源数据状态
- versionCode / versionName
- 安装包签名证书 SHA-256
- appId / 来源 UID
- 数据部件状态与权限采集状态
- 创建时间、大小和备份类型
- 对事务 undo 和事务期间产生的回滚备份，额外保存 `transactionRequestId`；崩溃恢复只能使用与当前事务 ID 完全匹配的回滚来源。

新写入的 manifest 使用 schema `5`。除旧版高级兼容恢复外，schema `5` 的状态判定、持久备份复用与切换返回点必须同时验证 packageName、backupKind、来源/目标用户和签名证书；任一字段不匹配时状态为 `UNKNOWN` 或拒绝恢复。

恢复时：

- packageName 或签名不匹配：禁止恢复。
- versionCode 不匹配：显示高风险确认，不得静默继续。
- 无身份信息的旧备份：标记“旧版未验证”，默认不允许快捷入口直接恢复；仅主 App 中的高级确认可继续。

## 9. UID/GID 与 SELinux

- UClone 工作区内的新 payload 使用 `tar -xopf -`，不继承 App UID/GID，应归属 `root:root`。
- 恢复到 App 私有目录后，必须修正为目标用户的 App UID/GID。
- `cache` / `code_cache` 组必须使用 Android 的 cache GID 映射，不得直接使用 `20000 + appId`。
- external data/media 与 OBB 的组必须分开建模，OBB 不得沿用 media GID。
- 恢复后必须执行目标路径的 SELinux context 修复。
- 所有递归 owner 修正使用不跟随符号链接的语义。

## 10. 权限与 AppOps

- 权限采集必须区分“采集成功且结果为空”与“采集失败”。
- 只有采集成功后才能重置或覆盖目标权限/AppOps。
- 采集失败时不得使用空文件清空目标状态。
- 默认 `MERGE` 模式下，源权限/AppOps 采集失败必须记录警告并跳过权限部件，已验证的数据部件仍可继续同步；不得因此把整项数据任务误报为失败。
- `EXACT` 模式或事务回滚所需的精确权限采集失败时，必须在目标数据修改前中止任务。
- schema `5` 的持久备份复用必须核验 `permissionCaptureState`：`valid` 必须有完整可验证的权限捕获目录，`excluded` 或 `skipped` 不得残留权限目录；字段缺失或不一致时拒绝复用。
- 恢复计数统计成功数，不得统计尝试数。
- 无法通过 shell 稳定恢复的特殊权限必须记录为需用户手动处理，不得静默跳过。

## 11. 工作区安全

默认根目录为 `/data/adb/uclone`。工作区路径安全规则必须用于所有读、写、移动和删除任务，而不只是 owner 修复：

- `rootDir` 不得为空、`/` 或非绝对路径。
- 所有路径经 `readlink -f` 后必须位于同一 canonical root 下。
- 自定义工作区必须包含由 Root 拥有、普通文件、非符号链接、权限 `0600` 的 `config/workspace.identity`。
- 受管顶层目录及其路径组件不得是符号链接。
- 默认与自定义工作区的根目录、`config` 和已存在的受管顶层目录必须由 `root` 拥有，且不得向 group 或 other 开放写权限。
- 任务不得跨文件系统扫描或修改文件。
- 复制、容量统计、递归清理与 owner 修复必须使用不跨挂载点的遍历；发现目标根路径为符号链接或存在跨文件系统目录时必须失败，不得继续复制或删除。
- 跨挂载检测必须先通过 `/proc/self/mountinfo` 识别工作区下的嵌套挂载；不得为每次复制或读取请求额外递归遍历整个工作区。真实遍历仍必须使用 `-xdev` 或等价限制。
- 工作目录不得包含空白字符或反斜杠，避免 mountinfo 的转义路径被错误解释。
- 任何 tar 复制管道必须保留源端和目标端失败状态；运行环境不支持 `pipefail` 时必须拒绝任务，不得把截断归档视为成功。
- 外部 requestId 终态索引无法读取或解析时必须拒绝新的外部任务，防止重放；只有用户明确确认并成功执行“清理日志”或“重置工作区”后才能清空并重建该索引。
- 环境检测必须 fail-closed：初始化或 guard 失败后不得继续 `mkdir`、写日志或执行业务命令。
- 任何删除脚本必须使用固定允许集和 canonical containment 双重验证。

## 12. 任务、事务与崩溃恢复

- 主 App、桌面快捷入口和模块请求统一经过 `ExternalActionService -> TaskCoordinator -> SyncEngine -> Root shell`。
- 同一时间只允许一个会修改数据或用户生命周期的 Root 任务。
- Root 互斥锁必须先生成完整所有权记录，再使用同文件系统硬链接原子抢占；阶段更新不得制造“锁存在但 owner 不可读”的窗口。旧版目录锁只能作为只读兼容输入并在确认进程失效后隔离。
- 工作区扫描也必须进入统一 admission，不得与备份、恢复、推送、删除或重置并发。
- 每个外部请求的 requestId 必须在 Hook、Provider、Service、任务、通知、日志和状态回执中一致。
- 每个 `SENT` 必须进入明确终态，不得永久停留在“已发送”。
- Root 事务必须持久化任务 ID、阶段、PID、boot ID、PID start ticks、回滚路径、已修改部件和 commit 状态。
- 进程死亡或重启后必须将上一个非终态任务归类为 `STILL_RUNNING`、`ORPHANED`、`FAILED_PROCESS_DIED` 或“待恢复”，并在解决前禁止新的修改任务。
- 任务成功标记只能在数据、owner/context、权限和状态标记都完成后提交。
- 任务失败后，真实恢复成功不得显示失败；真实恢复失败不得显示成功。

### 12.1 持久事务状态机

每个会修改真实 App 数据的任务必须在 `/data/adb/uclone/transactions/<requestId>/` 保存持久事务记录，至少包含：

- schemaVersion、requestId、operation、packageName。
- sourceUserId、targetUserId、选中的数据部件。
- 本次任务专属 undo 路径及其完整性状态。
- 当前事务阶段、已修改部件、commit 状态。
- 源/目标 App 执行门禁状态及门禁前 PackageManager 状态。
- startedAt、updatedAt、boot ID、Root PID 与 PID start ticks。

事务阶段至少区分：

`CREATED -> PRECHECKED -> SOURCE_PREPARED -> ROLLBACK_READY -> TARGET_GATED -> TARGET_MUTATING -> TARGET_WRITTEN -> METADATA_RESTORED -> VERIFIED -> COMMITTED -> CLEANED`

失败后只能进入：

`ROLLING_BACK -> ROLLED_BACK`，或 `RECOVERY_REQUIRED`。

强制规则：

- 每次进入不可逆阶段前，先将新状态写临时文件、同步落盘并原子替换，再修改真实数据。
- 只要 `targetMutated=true` 且 `committed=false`，目标 App 必须保持执行门禁。
- 恢复原始 stopped 状态必须通过 UClone 自身受限的 Root PackageManager Bridge 调用 `setPackageStoppedState`；门禁建立前先以原 stopped 值探测该能力，Bridge 不可用时必须在禁用 App 前拒绝任务。
- 自动回滚验证失败时进入 `RECOVERY_REQUIRED`，不得解除目标 App 门禁，不得允许同包新任务覆盖现场。
- App 启动或设备重启后必须扫描未完成事务。仍在运行的继续观察；Root 任务已死亡的自动回滚或进入恢复向导。
- 普通历史清理、日志清理和工作区维护不得删除 `RECOVERY_REQUIRED` 事务。

### 12.2 App 执行门禁

`am force-stop` 只能终止现有进程，不能独立承担事务期执行隔离。所有实时源采集和真实目标恢复必须建立按用户隔离的持久 App 执行门禁：

- 源 App：从门禁建立并验证后开始采集，直到完整源快照进入 UClone 临时目录并通过验证。
- 目标 App：从生成本次事务 undo 之前开始建立门禁，直到事务提交或自动回滚验证成功。
- 门禁顺序为：记录原始 PackageManager 状态 -> 禁用 App -> 重新读取验证 -> force-stop -> 验证进程消失 -> 执行数据操作。
- 必须保存并精确恢复 `DEFAULT`、`ENABLED`、`DISABLED`、`DISABLED_USER`、`DISABLED_UNTIL_USED` 以及原有 suspend 状态；禁止无条件执行 `pm enable`。
- 门禁建立或验证失败时，必须在修改任何目标数据前失败。
- Shared UID 默认拒绝严格恢复；只有确认冻结同 UID 全部普通 App 且不涉及系统关键包时才允许继续。
- UClone 自身、所有系统 App、Launcher、输入法、系统设置、电话及其他关键包不得进入数据门禁流程；除固定拒绝列表和系统包查询外，还必须按目标用户实时解析当前 HOME 与默认输入法。“高级系统 App 跨用户安装”只放开 package 启用，不放开系统 App 数据覆盖。
- 任务 ID、快照 ID、回滚 ID 与临时目录 ID 不得只使用秒级时间戳；必须包含 requestId 派生的合法后缀或同等的抗碰撞标识，且生成失败时安全终止任务。

### 12.3 合作式取消

- `PRECHECK` 与尚未生成真实目标变化的源准备阶段可以直接取消并清理临时文件。
- `ROLLBACK_BACKUP` 只能在安全检查点响应取消。
- `TARGET_MUTATING`、`AUTO_ROLLBACK` 与 `COMMIT` 阶段禁止普通取消直接强杀 Root Shell。
- 用户请求取消后写入事务目录中的取消标记，由 Root 脚本在安全检查点转入清理或自动回滚。
- 只有 Root 进程完全失去响应时才允许最终强杀；强杀后任务必须进入 `RECOVERY_REQUIRED`，目标 App 保持门禁。

## 13. 外部模块协议

调用链固定为：

`Launcher Hook -> ModuleRelayProvider -> one-shot PendingIntent.getForegroundService -> UClone ExternalActionService`

要求：

- Provider 使用 Binder caller UID 校验允许的系统 Launcher。
- PendingIntent 必须 one-shot、immutable，且实际启动身份是模块 APK UID。
- ExternalActionService 使用与 UClone/Module 同签名的 signature 权限保护。
- 模块默认禁止控制系统 App、UClone 与模块自身。
- 第一版 Hook 只注入 user0 普通 App 图标，分身图标、deep shortcut、文件夹、小部件和未知 userHandle 只记录不注入。
- Hook 层全部 try/catch，连续异常达阈值后自动禁用，不得导致 Launcher 循环崩溃。
- 任务成功后即使主 App UI 没有运行，也必须刷新桌面动态快捷方式状态；或使用不会过期的中性文案。

## 14. 删除与重置

- App 卸载不得自动删除 UClone 备份。
- 不增加 `PACKAGE_REMOVED` 自动清理机制。
- 删除主动快照、删除指定被动备份、清理日志和重置工作区是不同操作，必须用明确文案区分。
- 任何删除操作都必须经用户确认；重置工作区必须二次确认。
- 正常事务可按明确的保留策略轮换旧回滚点，但不得删除当前唯一可用返回点。
- 日志清理不得影响切换状态、备份索引或快捷按钮判断。

## 15. UI 与文案

- UI 采用紧凑的 Apple / iOS 液态玻璃视觉语言，但不牺牲 Android 可访问性、对比度和触控区域。
- 按钮必须有可见容器、一致高度、按压反馈和贴合轮廓的效果；可操作文本不得伪装成普通文本。
- 页面必须正确处理状态栏、刘海、底部导航和系统返回。二级页面必须有明确返回按钮并正确消费手势返回。
- 长路径使用可水平滚动的单行容器，不自动换行。
- 设置保存必须有 Toast/Snackbar 成功或失败反馈。
- 任务状态面向用户显示中文动作、阶段、数据量和耗时；内部 enum 只用于诊断详情。
- 数据页详情只提供恢复和删除等备份操作，不复用 App 详情的数据范围与同步操作。
- App 详情的多个工具采用“功能名称 + 精确说明 + 右侧执行按钮”的行布局，不使用连续的全宽大按钮。
- 首页收藏 App 一行一个，默认只显示一个主操作；推送作为明确的次级操作。

## 16. 可观测性与性能

- 任务日志必须包含 requestId、本地时间戳、任务类型、包名、Root 身份、阶段、退出码和总耗时。
- 性能指标至少包含扫描文件数、复制文件数、写入字节、Root 进程数、阶段耗时、目标 App 停机时间和临时空间峰值。
- 通知在阶段变化、终态或需要用户处理时立即更新；普通进度更新需节流。
- 成功任务可只保留摘要，失败任务保留完整日志。日志必须可由用户主动清理。
- 没有真机基准前，不引入增量复制、常驻 RootSession、rsync 或并行恢复。

## 17. 发布与验收门槛

- 代码修改完成后先运行 JVM 单元测试、lint 和 Debug 构建，再进行 Root/HyperOS 真机测试。
- 不得用 JVM 脚本字符串测试代替 Root 设备事务验证。
- 每个发布候选版本至少验证：主动备份、切换、还原、推送、自动回滚、分身启停、冷启动模块请求、两个用户的跨用户安装和工作区安全拒绝。
- 必须对 `SOURCE_PREPARE`、`ROLLBACK_BACKUP`、`TARGET_MUTATING`、`RESTORE_METADATA`、`VERIFY`、`COMMIT` 与 `AUTO_ROLLBACK` 分别执行杀 App 进程、杀 Root Shell和重启设备故障注入。
- 故障注入后只能得到四种可证明结果：目标未修改、事务完整提交、事务完整回滚、或 `RECOVERY_REQUIRED` 且目标 App 仍被冻结。
- 必须验证恢复关键窗口内从桌面图标、深链、旧通知、Widget、Job/Alarm 等入口均无法启动源/目标 App，并在事务结束后精确恢复原 PackageManager 状态。
- 快照文件数/字节数、目标 owner/GID、SELinux context、包签名和状态标记都必须纳入验收。
- 主 App 与模块必须同签名，发布前校验签名证书 SHA-256。
- 编译、安装、发布、删除 Release 或删除手机文件前，都必须获得用户明确确认。
- 所有 Root 删除操作必须在执行前通过对话框明确告知路径和影响范围。

## 18. 变更管理

- 新功能先更新本文档，再改代码。
- 实现与本文档冲突时，视为 Bug，不得以“已有行为”为理由保留。
- 每个高风险修复必须有对应的自动测试和真机验收项。
- 未完成真机验证的能力必须在发布说明中明确标注，不得宣称“完全稳定”。
